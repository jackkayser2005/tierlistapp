   207‚ÜíDeviations from spec (minor, all additive):
   208‚Üí1. **`presence:update` payload includes `host: string | null`** in addition to `{ members, seq }`. The spec literal text says `{ members: presenceList(room) }`, but the spec ALSO says "Broadcast presence:update so clients see the new host badge" ‚Äî so clients need the host id. Added `host` to the payload (and to the direct emit to the joiner). This is the only way clients can render the host badge after a promotion without an extra round-trip.
   209‚Üí2. **`vote:result` and `vote:state` payloads include `seq`** (spec says "Include seq in those payloads where it matters (board:update relay especially)" ‚Äî extended to vote events for consistency).
   210‚Üí3. **`activity:new` payload is `{ entry, seq }`** (spec says `{ entry }` ‚Äî added seq for monotonic ordering signal).
   211‚Üí4. **Room-switch (leaving previous rooms in room:join) scrubs the switching user's vote** on the old room's active vote (spec says "same as before, but also broadcast presence:update" ‚Äî added vote scrub for robustness, matching the disconnect behavior; otherwise a ghost vote would persist).
   212‚Üí5. **Room-switch does NOT push a "left" activity** (spec only specifies "left" on disconnect; preserved spec literally ‚Äî the user silently disappears from the old room's presence but no activity entry is logged for room switches).
   213‚Üí6. **`vote:state` to joiner on hydrate** uses current `room.seq` (no increment) ‚Äî same as the original hydration behavior; the spec's seq-increment rule applies to broadcasts, not direct hydration emits.
   214‚Üí7. **`board:update` no longer creates a room on the fly** if the room doesn't exist (original did; new version requires `room:join` first). This aligns with the identity-based join flow ‚Äî clients must join before updating.
